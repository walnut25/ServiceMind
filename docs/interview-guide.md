# Smart Service 求职讲解稿

## 简历项目描述

**Smart Service — AI 辅助企业服务台后端**

- 基于 Java 21、Spring Boot 3.5 与 MySQL 8.4 设计模块化单体服务台，完成身份、工单、知识库和
  RAG 问答 4 个业务模块、21 个 REST 接口。
- 实现 JWT + BCrypt + RBAC、请求人数据隔离、受控工单状态机、指派/评论/不可变审计轨迹、
  知识发布与全文检索，并通过乐观锁保护并发更新。
- 使用 Flyway、Testcontainers、Docker Compose 和 GitHub Actions 建立交付链路；33 项自动化
  测试覆盖领域规则、API 权限及真实 MySQL 行为，另提供可重复的端到端验收脚本。

不要在简历中写“微服务”或“已支撑高并发生产流量”。当前项目是有明确模块边界的模块化单体，
这是主动设计选择，不是缺陷。

## 60 秒项目介绍

> 这是一个面向企业内部 IT 支持场景的后端项目。用户可以提交工单，管理员把工单分配给坐席，
> 坐席按状态机处理并沉淀知识文章，AI 助手只基于已发布文章回答并返回来源。项目最值得讲的
> 不是 CRUD，而是权限和一致性：请求人只能看到自己的工单，对越权资源统一返回 404；每次
> 创建、指派、流转和评论都会和业务操作在同一事务里写入审计事件；知识文章通过发布状态控制
> 可见性，并用乐观锁防止并发覆盖。我用 Testcontainers 在真实 MySQL 8.4 上验证迁移、全文
> 索引和锁行为，再通过 Docker Compose 与端到端脚本验证完整用户旅程。目前是可部署、可演示
> 的后端 MVP，后续再根据规模演进向量检索、异步摄取和通知模块。

## 演示顺序

控制在 5 分钟内：

1. 展示 README 架构图和模块边界。
2. 运行 `scripts/smoke-test.ps1`，说明它不是简单健康检查，而是三种角色的完整业务旅程。
3. 打开 Swagger，展示登录、工单指派、状态流转和审计记录。
4. 展示知识文章发布和全文搜索。
5. 打开一项 Testcontainers 测试及 GitHub Actions，说明真实数据库验证和持续集成。

AI 没有配置真实密钥时，不要临场依赖外部调用；说明默认关闭是为了成本、稳定性和密钥安全，
然后展示本地测试覆盖的检索、提示词边界及降级行为。

## 高频追问与回答

### 为什么选择模块化单体，不直接做微服务？

当前团队和业务规模不需要承担服务发现、分布式事务、链路追踪和多套部署的成本。模块化单体让
一次事务可以同时完成工单更新和审计落库，同时用包结构、应用服务和仓储边界控制耦合。如果后续
某模块在吞吐量、团队归属或发布节奏上出现独立需求，可以先通过事件/outbox 解耦，再拆服务。

### 为什么请求人访问别人的工单返回 404，而不是 403？

返回 403 会泄露资源 ID 确实存在。服务层把“资源不存在”和“当前请求人不可见”统一映射为 404，
同时列表查询直接在数据库条件中限定请求人，避免先全量读取再做内存过滤。

### 如何保证审计事件不会和业务状态不一致？

状态修改与审计事件写入位于同一个事务中；任意一步失败都会整体回滚。因此不会出现状态已经变化
但没有审计记录，或有审计记录但状态没有变化的半完成情况。跨服务后会考虑事务 outbox。

### 为什么使用乐观锁？

工单和知识文章的编辑冲突通常不是持续热点，乐观锁比数据库长事务锁更简单、吞吐更好。实体版本
不匹配时明确返回冲突，让调用方重新读取后决定如何合并，而不是静默覆盖别人的修改。

### RAG 如何减少幻觉和提示词注入？

系统只检索已发布文章，限制上下文条数与长度，把检索内容标记为不可信数据，并要求模型仅依据
上下文回答和返回来源。没有匹配文章时不调用模型，直接返回未找到可靠依据。它能降低风险但不能
“彻底消灭幻觉”，生产环境还应增加离线评测、引用校验、内容权限和安全过滤。

### 为什么集成测试一定用 Testcontainers？

H2 无法可靠代表 MySQL 的全文索引、SQL 方言、迁移和锁语义。Testcontainers 每次启动一次性
MySQL 8.4，测试和部署使用同类数据库；代价是速度更慢，所以把 30 项快速测试与 3 项数据库测试
分层执行。

### 用户禁用后，旧 JWT 会立即失效吗？

当前资源服务器验证 JWT 签名和到期时间，禁用会阻止新登录，但已签发 token 不维护集中式撤销
列表。这是当前 MVP 的明确边界。生产化可使用短期 access token + refresh token 轮换、用户版本
声明或撤销缓存实现近实时失效。

### 全文搜索为什么不是向量数据库？

第一版用 MySQL FULLTEXT 得到零额外基础设施、可事务发布、可测试的关键词检索，足以证明完整
RAG 链路。语义召回成为实际瓶颈后，再增加分块、embedding、混合检索与重排；在没有评测数据前
直接引入向量库只会增加复杂度。

## 当前边界和下一步

可以坦诚说明以下内容尚未生产化：

- access token 即时撤销、refresh token、限流与企业 SSO
- 文档分块、向量/混合检索、答案质量评测
- 通知、SLA、报表和异步任务
- Kubernetes、集中式密钥管理、生产可观测告警

如果只选一个下一步，优先做“知识文档异步摄取 + 可量化 RAG 评测”；它最能延伸当前项目的 AI
主题。若目标岗位偏传统 Java 后端，则优先补 Redis 限流、outbox 事件和并发压测。
